# 開発ガイドライン

## 設計プロセス

### 1. ADR（Architecture Decision Record）
- 新機能の実装前に必ずADRを作成
- `docs/ADR/` 配下に配置
- テンプレート: `docs/adr_template.md` を使用
- 以下の項目を必ず含める：
  - コンテキストと要件
  - 決定事項と根拠
  - 代替案の検討
  - 影響範囲
  - 妥協点・技術的負債

### 2. ドメイン知識の文書化
- `docs/domain/` 配下に配置
- 業務知識・用語の定義
- 制約・ルール
- 将来の拡張可能性
- 参考文献・出典

### 3. コンポーネントと言語間の一貫性
- Python/TypeScript間でのモデル定義の同期
- 共通の型定義・バリデーションルール
- コードジェネレータの活用（必要に応じて）
- スキーマ定義の一元管理

### 4. フロントエンドコンポーネント設計
- プロジェクト全体で一貫したexport/import方式の採用
  - 初期設計時に明確な方針決定（named exportまたはdefault export）
  - コンポーネントとテストで同一の方式を使用
  - 方針変更時は一括で修正
- コンポーネントの責務を明確に分離
  - 単一責任の原則に従う
  - テスト可能性を考慮した設計
  - プロップスとステートの適切な使用
- Routerとテストの統合
  - Router周りのテスト実装ベストプラクティス
  - テスト用のラッパーコンポーネント活用
  - メモ化による最適化

## 実装プロセス

### 1. テスト駆動開発（TDD）
- テストファースト原則
- Red-Green-Refactorサイクルの遵守
- テストケースの優先順位：
  1. バリデーションテスト
  2. コアロジックテスト
  3. エッジケーステスト
  4. 異常系テスト

### 2. テストの一貫性と品質
- Python（pytest）とTypeScript（Jest）で同等のテストケース
- テストケース命名規則の統一
- テストデータの共有
- カバレッジ目標：80%以上
- テストのベストプラクティス
  - 実装前のテストケース設計レビュー
  - 非同期処理（waitFor等）の適切な使用
  - Router/Reduxのテスト手法標準化
  - モックの使用指針
- エッジケースの網羅
  - 境界値テスト
  - エラー処理テスト
  - 非同期処理のエッジケース
- テストデータ管理
  - 再利用可能なフィクスチャ
  - テストデータファクトリの活用
  - スナップショットテストの活用

### 3. バリデーション
- 入力値の検証（Pydantic, Zod等）
- ビジネスルールの検証
- エラーメッセージの一貫性
- 国際化対応

### 2. CI/CDツールチェーン
- **検証コマンドリファレンス**

  **フロントエンド検証**
  ```bash
  # 依存関係の更新と整合性確認
  cd frontend
  rm -rf node_modules package-lock.json
  npm install --legacy-peer-deps

  # ESLintによるコード品質チェック
  npm run lint
  # または詳細なレポート出力
  npx eslint . --ext .js,.jsx,.ts,.tsx --report-unused-disable-directives

  # Prettierによるコードフォーマットチェック
  npx prettier --check "src/**/*.{ts,tsx}"
  # フォーマット適用
  npx prettier --write "src/**/*.{ts,tsx}"

  # TypeScript型チェック
  npm run type-check
  # または詳細オプション付き
  npx tsc --noEmit --incremental false

  # テスト実行
  npm test
  # カバレッジレポート付き
  npm test -- --coverage
  ```

  **バックエンド検証**
  ```bash
  # 仮想環境の再作成と依存関係の更新
  cd backend
  source .venv/bin/activate  # Windows WSLの場合
  uv pip install -e ".[dev,test]"
  uv sync --all-extras

  # ruffによるコード品質チェック
  ruff check .
  # 自動修正
  ruff check . --fix

  # ruffによるフォーマットチェック
  ruff format --check .
  # フォーマット適用
  ruff format .

  # mypyによる型チェック（詳細出力）
  mypy . --show-error-codes --pretty

  # pytestによるテスト実行
  pytest
  # カバレッジレポート付き
  pytest --cov=. --cov-report=term-missing
  # 詳細出力とログ表示
  pytest -v --log-cli-level=INFO
  ```

  **WSL環境でのデバッグ用追加コマンド**
  ```bash
  # Pythonバッファリング無効化（出力問題対策）
  python -u -m pytest

  # ログファイルへの出力
  pytest --log-file=test.log

  # エラー発生時のPDB起動
  pytest --pdb

  # WSLでのパス問題のデバッグ
  realpath .  # 現在の絶対パスを表示
  pwd  # 現在のディレクトリを表示
  ```

- **CI実行前のローカルチェックリスト**
  1. 最新のmainブランチを取り込んでいることを確認
  2. 依存関係を最新化
  3. 全検証コマンドを順次実行
  4. すべてのチェックがパスすることを確認
  5. コミット前にpre-commitフックが正常に動作することを確認

## バージョン管理とコミット規約

### 1. コミットメッセージ規約
- **メッセージフォーマット**
  - 「動詞: 内容」の形式を使用
  - 1行50文字以内で簡潔に
  - 変更内容を明確に記載
  - 例：
    - `feat: add task list pagination`
    - `fix: resolve router testing issues`
    - `docs: update development guidelines`

### 2. コミット粒度
- **単一責任の原則**
  - 1つの論理的な変更につき1コミット
  - 関連する変更のみをグループ化
  - 不要なファイルの混入を防止
- **レビュー容易性**
  - 適切な粒度での分割
  - 変更の意図が明確
  - コンテキストの維持

### 3. ブランチ戦略
- **ブランチ命名規則**
  - 機能追加：`feature/機能名`
  - バグ修正：`fix/問題の概要`
  - ドキュメント：`docs/更新内容`
- **マージ前の品質確保**
  - ローカルテストの完全実行
  - Lint/Format/型チェックの確認
  - コンフリクト解消の確認
